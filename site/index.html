Real example

"Should we migrate from REST to GraphQL?"

A complete wheat sprint, from question to recommendation. Every step shows the actual commands and output.

1
$ npx @grainulator/wheat init

Start the sprint

You type your question: "Should we migrate from REST to GraphQL?" Wheat asks for the audience (backend team) and constraints (must support existing mobile clients), then creates claims.json and wheat.config.json.

2
/research "GraphQL performance characteristics"

Gather evidence

Wheat researches your codebase and the topic, producing typed claims with evidence tiers.

r001 [factual|documented] GraphQL eliminates over-fetching for mobile clients, reducing payload size 40-60% r002 [factual|web] GraphQL N+1 query problem requires DataLoader or equivalent batching layer r003 [risk|documented] Existing REST caching (CDN, HTTP) does not transfer to GraphQL without custom solutions
3
/prototype

Build and measure

Wheat builds a working proof-of-concept and tests the claims. Prototype findings get the tested evidence tier.

p001 [estimate|tested] Migration effort: ~3 weeks for core endpoints, 2 weeks for mobile client updates p002 [factual|tested] DataLoader resolves N+1; p95 latency within 12% of REST equivalent
4
/challenge r001

Stress-test the claims

Adversarial review. Does the evidence actually support the conclusion?

x001 [risk|documented] 40-60% payload reduction assumes worst-case REST over-fetching; our actual endpoints are already field-filtered -- real reduction ~15-25%
5
/brief

Get the answer

The compiler resolves conflicts (higher evidence tiers win) and produces a self-contained HTML brief.

Recommendation: Proceed with GraphQL for new endpoints only. Migrate existing REST endpoints opportunistically. Payload benefits are real but smaller than initially estimated (15-25% vs. 40-60%). The caching gap (r003) needs a solution first. 14 claims | 3 conflicts resolved | evidence: 2 tested, 8 documented, 4 web -> output/brief.html (self-contained, send to stakeholders)
Key features

Built for real research

Everything you need to go from question to decision. Nothing you don't.

Evidence-Graded Claims

Every finding is typed (factual, risk, estimate, recommendation) with a 5-tier evidence scale from "stated" to "production".

Compiler Validation

Claims are compiled before any output. The compiler catches conflicts, flags weak evidence, and resolves disputes by tier.

16 Slash Commands

init, research, prototype, challenge, witness, blind-spot, brief, present, status, feedback, handoff, merge, replay, calibrate, resolve, evaluate.

Git Audit Trail

Every claim auto-commits. Run git log --oneline claims.json for the full history of your research sprint.

Self-Contained Output

Briefs, presentations, and dashboards are single HTML files with inline CSS/JS. Send them to anyone, no hosting needed.

Language Agnostic

Works in any repo. JavaScript, Python, Scala, Rust, Flutter -- if Claude Code can read it, wheat can research it.
