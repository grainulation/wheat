What is wheat?

wheat is a research tool for developers who need to make technical decisions. You start a "sprint" by asking a question — like "Should we migrate to GraphQL?" — and wheat guides you through gathering evidence, testing assumptions, and resolving conflicts.

Every finding is tracked as a typed "claim" with a confidence level. When you are done, wheat compiles everything into a decision brief you can send to your team. It runs inside Claude Code, so you use natural language and slash commands — no new UI to learn.

Real example

"Should we migrate from REST to GraphQL?"

A team needs to decide whether to invest 3-5 weeks in a GraphQL migration. They use wheat to research it properly instead of debating in Slack.

1
$ npx @grainulation/wheat init

Start the sprint

The team lead types the question: "Should we migrate from REST to GraphQL?" wheat asks for audience (backend team), constraints (must support existing mobile clients), and creates the sprint structure.

2
/research "GraphQL performance characteristics"

Gather evidence

wheat researches the codebase and the topic, producing typed claims. Each claim has a type (factual, risk, estimate) and an evidence tier (how well-verified it is).

r001 [factual|documented] GraphQL eliminates over-fetching for mobile clients, reducing payload size 40-60% r002 [factual|web] GraphQL N+1 query problem requires DataLoader or equivalent batching layer r003 [risk|documented] Existing REST caching (CDN, HTTP) does not transfer to GraphQL without custom solutions
3
/prototype

Build and measure

wheat builds a working proof-of-concept and benchmarks it. Findings from prototypes get the highest evidence tier — "tested" — because they come from real measurements, not blog posts.

p001 [estimate|tested] Migration effort: ~3 weeks for core endpoints, 2 weeks for mobile client updates p002 [factual|tested] DataLoader resolves N+1; p95 latency within 12% of REST equivalent
4
/challenge r001

Stress-test the claims

This is where wheat shines. The adversarial review catches that the "40-60% payload reduction" was based on worst-case assumptions — the real number is lower because the team already field-filters their REST endpoints.

x001 [risk|documented] 40-60% payload reduction assumes worst-case REST over-fetching; our actual endpoints are already field-filtered — real reduction ~15-25%
5
/brief

Get the answer

The compiler resolves conflicts (higher evidence tiers win), validates all claims, and produces a self-contained HTML decision document the team lead can share in Slack or attach to a Jira ticket.

Recommendation: Proceed with GraphQL for new endpoints only. Migrate existing REST endpoints opportunistically. Payload benefits are real but smaller than initially estimated (15-25% vs. 40-60%). The caching gap (r003) needs a solution first. 14 claims | 3 conflicts resolved | evidence: 2 tested, 8 documented, 4 web -> output/brief.html (self-contained, send to stakeholders)

The outcome: Instead of a 45-minute meeting where the loudest voice wins, the team has a 14-claim brief with evidence tiers, conflict resolution, and a clear recommendation. The decision is traceable — anyone can run git log claims.json to see how they got there.

Key features

Built for real research

Everything you need to go from question to decision. Nothing you don't.

Evidence-Graded Claims

Every finding is typed (factual, risk, estimate, recommendation) with a 5-tier evidence scale from "stated" to "production-measured". silo stores proven claims as reusable packs for future sprints.

Compiler Validation

Claims are compiled before any output. The compiler catches contradictions, flags weak evidence, and resolves disputes by tier.

16 Slash Commands

/init, /research, /prototype, /challenge, /witness, /blind-spot, /brief, /present, /status, /feedback, /handoff, /merge, /replay, /calibrate, /resolve, /evaluate.

Git Audit Trail

Every claim auto-commits. Run git log --oneline claims.json for the full history of your research sprint. farmer streams these tool calls to your phone for real-time approval.

Self-Contained Output

Briefs, presentations, and dashboards are single HTML files with inline CSS/JS. Send them to anyone — no hosting needed. Use mill to export to PDF, CSV, or static sites.

Language Agnostic

Any repo, any stack, any language. If Claude Code can read it, wheat can research it.

Frequently asked questions

Do I need Node.js?

Yes, Node 18 or later. But your project can use any language or framework — wheat works in any repo regardless of stack.

Does this require Claude Code?

Yes. wheat runs as a set of slash commands inside Claude Code. It uses Claude's ability to read your codebase, search the web, and reason about evidence.

How does this compare to just asking Claude a question?

Asking Claude gives you one answer. wheat gives you a structured research process — multiple rounds of evidence gathering, adversarial challenge, prototype validation, and conflict resolution. The output is an auditable decision brief, not a chat message.

How long does a sprint take?

A simple sprint can be done in 10-15 minutes. Complex questions with prototyping and multiple rounds of challenge might take an hour or spread across a few sessions. When running parallel sprints, orchard tracks dependencies between them.
