Real example

"Should we migrate from REST to GraphQL?"

Here is what a wheat sprint looks like from start to finish. Each step produces typed, evidence-graded claims that flow through the compiler.

1
/init "Should we migrate from REST to GraphQL?"

Define the sprint

Sets the question, audience (backend team), and constraints (must support existing mobile clients). Creates claims.json and wheat.config.json.

2
/research "GraphQL performance characteristics"

Gather evidence

Investigates the topic, producing claims with evidence tiers. Each claim is typed and traceable.

r001 [factual|documented] GraphQL eliminates over-fetching for mobile clients, reducing payload size 40-60% r002 [factual|web] GraphQL N+1 query problem requires DataLoader or equivalent batching layer r003 [risk|documented] Existing REST caching (CDN, HTTP) does not transfer to GraphQL without custom solutions
3
/prototype

Build and measure

Creates a working proof-of-concept. Prototype findings get the tested evidence tier -- the second highest.

p001 [estimate|tested] Migration effort: ~3 weeks for core endpoints, 2 weeks for mobile client updates p002 [factual|tested] DataLoader resolves N+1 for our schema; p95 latency within 12% of REST equivalent
4
/challenge r001

Stress-test claims

Adversarial review of individual claims. Does the evidence actually support the conclusion? What are we not seeing?

x001 [risk|documented] 40-60% payload reduction assumes worst-case REST over-fetching; our actual endpoints are already field-filtered -- real reduction likely 15-25%
5
/brief

Get the answer

The compiler resolves conflicts (higher evidence tiers win), validates the claim set, and produces a self-contained HTML brief you can send to stakeholders.

Recommendation: Proceed with GraphQL for new endpoints only. Migrate existing REST endpoints opportunistically. The payload benefits are real but smaller than initially estimated (15-25% vs. 40-60%). The caching gap (r003) needs a solution before full migration. 14 claims | 3 conflicts resolved | evidence: 2 tested, 8 documented, 4 web
Get started

Install in 30 seconds

Wheat runs inside Claude Code. No framework, no build step. Your target project does not need Node.

npx (recommended)

Run directly without installing. Works in any repo.

npx @grainulator/wheat init

Global install

Install once, use everywhere.

npm i -g @grainulator/wheat
# then in any repo:
wheat init

Clone and link

For contributors or custom setups.

git clone https://github.com/aid-ninja/wheat
cd wheat
# install into your project:
node wheat-init.js /path/to/repo

Under the hood

Claims-driven research

Every finding in a wheat sprint is a typed claim. Claims have a type (factual, risk, estimate, recommendation) and an evidence tier (stated, web, documented, tested, production).

The wheat compiler validates the full claim set before any output artifact is produced. It resolves conflicts by evidence tier, flags unsupported recommendations, and catches claims that reference nonexistent sources.

Because every claim auto-commits to git, git log --oneline claims.json gives you the complete audit trail of your research sprint.

{ "id": "r001", "type": "factual", "claim": "GraphQL reduces mobile payload 40-60%", "evidence": "documented", "source": "graphql.org/learn/best-practices", "confidence": 0.7, "superseded_by": "x001" }

All commands run inside Claude Code as slash commands or natural language.

/initBootstrap a sprint -- question, audience, constraints
/researchDeep-dive into a topic, produce claims
/prototypeBuild a proof-of-concept, tested evidence
/challengeAdversarial review of a specific claim
/witnessCorroborate a claim against an external source
/blind-spotFind structural gaps in research coverage
/evaluateScore and compare options
/briefGenerate a decision document from compiled claims
/presentCreate stakeholder presentation
/statusSprint dashboard snapshot
/feedbackIncorporate stakeholder input
/handoffKnowledge transfer document for successors
/mergeCombine findings from multiple sprints
/replayVisualize sprint history and evolution
/calibrateScore past predictions against actual outcomes
/resolveAdjudicate conflicts between claims
